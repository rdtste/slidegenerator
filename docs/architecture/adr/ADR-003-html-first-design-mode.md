# ADR-003: HTML-First Rendering with Visual Preview for Design Mode

**Status:** Proposed
**Date:** 2026-04-02

## Context

The V2 pipeline renders directly to PPTX via python-pptx with fixed coordinate blueprints. There is no visual preview before export -- the first time anyone "sees" the slide is in the post-render Vision Review (Stage 8). This means bad composition decisions are baked into the PPTX before they can be caught.

Additionally, python-pptx TextBox placement is brittle. Font metrics differ between python-pptx's internal calculations and actual PowerPoint rendering, causing text overflow, misalignment, and inconsistent spacing. The coordinate-based blueprint system (`pptx-service/app/layouts/`) requires manual pixel-level tuning for every layout variant, which does not scale.

The Vision Review agent (Stage 8) can detect problems after rendering, but its fixes are limited to minor adjustments (font size, spacing). Fundamental composition problems -- wrong layout choice, poor visual hierarchy, unbalanced content distribution -- cannot be fixed at that stage.

## Decision

In Design Mode, compose slides as HTML using React and Tailwind CSS first. Use this HTML rendering as the basis for both visual preview and quality validation.

**Rendering pipeline for Design Mode:**

1. **HTML Composition:** Each slide is rendered as a React component with Tailwind CSS. The component receives structured slide content and produces a 16:9 viewport (1280x720 or 1920x1080).
2. **Browser Preview:** Puppeteer or Playwright renders the HTML to a viewport and takes a screenshot.
3. **Visual Quality Gate:** The screenshot is sent to the Vision Review agent. Problems are caught before PPTX export.
4. **PPTX Export:** After passing the quality gate, the slide is exported to PPTX via PptxGenJS (Node.js-native PPTX generation) or a dom-to-pptx bridge.

**Technology choices:**
- **React + Tailwind CSS** for slide composition -- modern, widely understood, excellent layout primitives (flexbox, grid, gap).
- **Puppeteer/Playwright** for screenshot generation -- headless Chrome provides pixel-perfect rendering.
- **PptxGenJS** for native PPTX generation -- JavaScript library that generates PPTX natively, avoiding python-pptx font metric issues.
- **dom-to-pptx bridge (optional)** for converting HTML layout coordinates to PPTX shapes -- experimental, may be used if PptxGenJS mapping is insufficient.

**Service architecture:**
A new Node.js/TypeScript rendering service sits alongside the Python orchestrator. The orchestrator sends structured slide content to the rendering service, which returns screenshots and PPTX bytes.

## Consequences

- Much better visual fidelity -- what you preview is what you get. HTML rendering in Chrome matches how users expect slides to look.
- The Visual Quality Gate can catch composition problems before PPTX export, enabling re-composition rather than post-hoc fixes.
- Requires a Node.js/TypeScript rendering service alongside the Python orchestrator, adding operational complexity (one more container to deploy and monitor).
- Tailwind CSS provides a rich design vocabulary (shadows, gradients, rounded corners, responsive spacing) that is difficult to replicate with python-pptx coordinate math.
- PPTX export fidelity depends on how well PptxGenJS can reproduce the HTML layout. Some CSS features (gradients, shadows, complex borders) may not translate perfectly to PPTX.
- The rendering service needs a headless browser (Chromium), which increases container size and memory usage.
- Design Mode slides are no longer generated by python-pptx, so the existing V2 renderer code becomes Template Mode only.

## Alternatives Considered

- **Improve python-pptx rendering (current approach):** Continue refining coordinate blueprints and font metric calculations. Rejected because the fundamental mismatch between python-pptx metrics and PowerPoint rendering cannot be fully resolved without a visual rendering engine.
- **SVG-based rendering:** Compose slides as SVG and convert to PPTX. Rejected because SVG layout primitives (no flexbox, no grid) make responsive composition harder, and SVG-to-PPTX conversion is poorly supported.
- **PowerPoint COM automation:** Use PowerPoint's own rendering engine via COM/AppleScript. Rejected because it requires a PowerPoint installation, is platform-dependent, and does not scale in containerized environments.
- **LibreOffice rendering:** Use LibreOffice Impress to render previews. Already used in the QA loop, but LibreOffice rendering fidelity differs from PowerPoint, and it cannot be used for composition (only for preview of existing PPTX files).
